# 로봇 플릿 관제 대시보드 — UX 설계 문서

> 작업 지시서(WORK_INSTRUCTIONS.md)의 화면/인터랙션을 어떻게 만들지 결정하는 문서.
> 핵심 전제: 이건 일반 웹앱이 아니라 **종일 켜놓고 보는 관제(monitoring) 화면**이다. UX 원칙이 다르다.

---

## 1. 사용자 & 사용 맥락

- **사용자**: 관제 운영자. 수백~수천 대 로봇의 상태를 동시에 감시
- **사용 맥락**: 큰 모니터에 화면을 띄워두고 **하루 종일 본다**. 능동적으로 클릭하는 시간보다 **흘끗 보는(glance)** 시간이 훨씬 길다
- **운영자가 실제로 하는 일**: "지금 문제 있는 놈 있나?"를 1초 안에 판단 → 있으면 그놈만 파고듦
- **설계 함의**: 평상시엔 조용하고, 이상이 생기면 즉시 눈에 들어와야 한다

---

## 2. 핵심 UX 원칙 (관제 화면 특화)

1. **Glanceability(흘끗 가독성) 최우선** — 전체 플릿 건강 상태를 2초 안에 파악. 숫자 읽기 전에 색으로 먼저 감 잡히게
2. **정상은 조용히, 이상은 크게** — 모든 게 깜빡이면 아무것도 안 보인다(alarm fatigue). 정상 상태는 차분한 색, 에러만 부각
3. **다크 테마 기본** — 종일 보는 화면이라 눈 피로 최소화. 관제실 표준이기도 함
4. **레이아웃 안정성** — 실시간 업데이트가 화면을 흔들면 안 됨. 읽던 행이 갑자기 순서 바뀌거나 사라지면 못 읽는다
5. **밀도 높되 정렬** — 정보는 빽빽해도 되지만, 정렬과 그룹핑으로 스캔 가능하게

---

## 3. 정보 위계 (위 → 아래 = 중요 → 상세)

1. **전체 요약** (가장 위, 가장 큼): 에러 N대 / 오프라인 N대 / 충전 N대 / 가동 N대 / 전체
2. **이상 항목 우선 노출**: 에러·오프라인 로봇이 기본 정렬 상단에 오도록 옵션 제공
3. **개별 로봇 행**: 이름·상태·배터리·현재작업
4. **디테일**: 선택했을 때만 (평소엔 숨김)

→ 운영자 시선 흐름: 상단 요약에서 "에러 3" 보임 → 리스트에서 에러 3대 찾음 → 클릭해서 상세

## 4. 레이아웃

```
┌─────────────────────────────────────────────┐
│  [전체 1000] [가동 820] [충전 90] [⚠에러 3] [오프 87] │  ← 요약 바 (고정)
├──────────────────────────────┬──────────────┤
│  🔍검색  [상태 필터 토글]       │              │
│  ┌────────────────────────┐  │   디테일      │
│  │ RBT-0042  ●moving  87% │  │   패널        │
│  │ RBT-0043  ●error   12% │  │  (선택 시만)  │
│  │ RBT-0044  ●idle    64% │  │              │
│  │ ... (가상화 스크롤)      │  │              │
│  └────────────────────────┘  │              │
└──────────────────────────────┴──────────────┘
```

- **요약 바**: 상단 고정. 스크롤해도 항상 보임
- **리스트**: 메인 영역. react-window 가상화
- **디테일 패널**: 우측 슬라이드. 선택 없으면 숨김(리스트가 전체 폭 사용)

---

## 5. 상태 색상 체계 (일관성이 생명)

| 상태 | 색 | 의도 |
|---|---|---|
| moving | 청록/녹색 | 정상 가동 — 안심 |
| idle | 중립 회색 | 대기 — 무채색, 시선 안 끎 |
| charging | 노랑/앰버 | 충전중 — 부드러운 주의 |
| error | **빨강** | 즉시 부각. 화면에서 가장 강한 색 |
| offline | 어두운 회색 | 연결 끊김 — 흐릿하게 |

- **빨강은 오직 error에만.** 다른 데 빨강 쓰면 에러가 안 보인다
- 색만으로 구분하지 말고 **상태 텍스트/아이콘 병기** (6번 접근성 참고)

### 배터리
- `> 20%`: 정상색(녹색~중립)
- `10~20%`: 앰버 (warning)
- `< 10%`: 빨강 + **느린 점멸** (critical)

---

## 6. 핵심 인터랙션

- **행 클릭** → 디테일 패널 열림. 같은 행 다시 클릭 또는 X → 닫힘
- **검색** → 이름으로 즉시 필터(디바운스 200ms)
- **상태 필터 토글** → "에러만" "오프라인만" 등 빠른 격리. 운영자가 제일 자주 쓸 기능
- **정렬** → 상태 / 배터리 / 이름. 단 **자동 재정렬 금지**(아래 7번)

---

## 7. 실시간 업데이트 동작 (이 도메인의 핵심)

평범한 대시보드와 가장 크게 갈리는 지점. 잘못하면 화면이 정신없어진다.

- **레이아웃 점프 금지**: 1초마다 값이 바뀌어도 행 순서·위치는 고정. 정렬은 **사용자가 명시적으로 누를 때만** 바뀐다. (읽고 있는데 행이 위로 튀어오르면 최악)
- **변경분만 미세 하이라이트**: 상태가 바뀐 행은 배경을 0.5초간 살짝 밝혔다가 페이드. "방금 바뀜"을 인지시키되 거슬리지 않게
- **숫자는 부드럽게 교체**: 카운트업 애니메이션 같은 건 빼고, 값만 조용히 갱신. 관제는 화려함보다 정확성
- **에러 발생 순간만 강조**: idle→error 전환 시 요약 바의 에러 카운트가 오르고 해당 행이 빨갛게. 단 **소리·전체화면 알림은 절제** — 데모에선 시각적 강조까지만
- **업데이트 일시정지 토글**(선택): 특정 로봇을 자세히 볼 때 화면 갱신을 잠깐 멈추는 기능. 운영자 배려 디테일

---

## 8. 화면 상태(State) 처리

- **로딩**: 서버 컴포넌트에서 초기 목록을 받아오므로 첫 화면은 스피너 없이 바로 채워짐. 폴링 갱신은 백그라운드(인디케이터만)
- **빈 상태**: 필터 결과 0건 → "조건에 맞는 로봇이 없습니다" + 필터 초기화 버튼
- **에러 상태**: 폴링 실패 시 상단에 "연결 끊김, 재시도 중..." 배너. 마지막 데이터는 유지하되 "N초 전 데이터" 표시(stale 표시 중요 — 관제에서 오래된 데이터를 최신으로 착각하면 위험)

---

## 9. 반응형

- **데스크톱 우선**: 관제는 큰 화면이 기본. 넓은 폭에서 리스트+디테일 2단
- **모바일**: 요약 바(가로 스크롤 가능) + 리스트 1단. 디테일은 풀스크린 오버레이로
- 모바일은 "이동 중 빠른 확인"용 보조 화면으로 취급

---

## 10. 접근성

- **색에만 의존하지 않기**: 상태는 색 + 텍스트(또는 아이콘 모양)로 이중 표현. 적록색약 운영자도 error/moving 구분 가능해야 함
- 상태 배지에 색 + 점 모양/패턴 차이 또는 라벨 병기
- 충분한 대비(다크 배경 위 텍스트 WCAG AA 이상)
- 키보드: 행 포커스 이동 + Enter로 디테일 열기
